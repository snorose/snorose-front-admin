# /commit

변경사항을 논리 단위로 자동 분석해 커밋을 나눠 수행한다. 직접 스테이징할 필요 없다.

## 사전 검사

1. `git branch --show-current`로 현재 브랜치 확인
   - 브랜치가 `dev`이면 AskUserQuestion 툴로 경고 후 확인:
     질문: "⚠️ 현재 dev 브랜치입니다. 정말 여기에 커밋할까요?"
     선택지: 계속 진행 / 취소
     - "취소" 선택 시 중단

2. `git status`와 `git diff`로 staged + unstaged 전체 변경사항 파악
   - 변경사항이 전혀 없으면: `커밋할 변경사항이 없습니다.` 출력 후 종료
   - `.env`, 비밀번호, API 키 패턴이 감지되면 즉시 중단하고 사용자에게 알린다

3. 변경된 파일들을 논리적 단위로 그룹핑한다
   - 서로 다른 목적의 변경은 별도 커밋으로 분리 (예: 기능 추가 + 설정 변경)
   - 같은 목적이면 하나의 커밋으로 묶기
   - 판단 기준: 변경 이유가 다르면 나눈다

## 커밋 메시지 작성 규칙

**.husky/commit-msg 정규식**: `^((docs|feat|fix|modify|refactor|style|test|chore): .+|(Merge|Revert|Initial|v[0-9]).*)$`

허용 태그와 용도:

- `feat` — 새로운 기능 추가
- `fix` — 버그, 오류 수정
- `modify` — 기능 변경, 오타 등으로 인한 코드 수정
- `refactor` — 기능 변경 없는 유지보수 (구조 변경, 불필요한 리소스 삭제, 이름 변경)
- `style` — UI/UX, 스타일링 등 디자인 관련 작업
- `docs` — README, WIKI 등 문서 수정
- `test` — 테스트 코드 추가 또는 수정
- `chore` — 빌드, 패키지 관리, 주석 등 기타 작업

형식: `tag: 한국어 설명`

예시:

- `feat: 사용자 검색 필터 추가`
- `fix: 페이지 이동 시 스크롤 위치 초기화 안 되는 버그 수정`
- `refactor: 공통 훅으로 중복 로직 분리`
- `chore: ESLint 규칙 업데이트`

메시지 작성 기준:

- 변경 대상과 핵심 의도가 드러나도록 구체적으로
- 여러 변경이 섞여 있으면 가장 중요한 것 하나를 제목으로, 나머지는 필요시 본문에
- `수정`, `변경`처럼 넓은 표현 단독 사용 금지

## 실행 확인

커밋 계획을 텍스트로 먼저 출력한 뒤, AskUserQuestion 툴로 인터랙티브 선택지를 제시한다:

질문: "이 계획으로 진행할까요?"
선택지:

- 전체 실행 (기본값)
- 취소
- 계획 수정

- 방향키로 선택, Enter로 확정
- "전체 실행" 선택 시 각 커밋을 순서대로 자동 실행 (중간 재확인 없음)
- "계획 수정" 선택 시 수정 요청을 텍스트로 입력받아 계획 재작성 후 다시 확인
- push는 사용자가 명시적으로 요청한 경우에만 수행

## 금지

- `--no-verify` 플래그 사용 금지
- 사용자 확인 없이 push 금지
