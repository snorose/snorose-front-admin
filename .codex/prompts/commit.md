현재 저장소의 `.husky/commit-msg`를 읽고 해당 커밋 메시지 규칙을 사용합니다.

## 분석 범위

- 스테이징된 변경사항만 분석합니다.
- 판단 기준은 `git diff --cached --stat`와 `git diff --cached`입니다.
- 스테이징된 변경사항이 없다면 정확히 다음 문구만 출력합니다:

스테이징된 변경사항이 없습니다.

## 커밋 메시지 작성

스테이징된 변경사항이 있다면 `.husky/commit-msg`를 만족하는 커밋 메시지 한 줄을 작성합니다.

- `tag: message` 형식을 사용합니다.
- `.husky/commit-msg`에서 허용하는 태그만 사용합니다.
- 스테이징된 변경사항에 가장 정확히 맞는 태그를 선택합니다.
- 메시지는 간결하고 구체적으로 작성합니다.
- 이 저장소의 커밋 기록에 맞춰 메시지 본문은 한국어를 우선 사용합니다.
- 사용자가 `FOCUS`를 제공했다면 추가 맥락으로만 사용합니다: $FOCUS

## 실행 방식 확인

`git commit` 또는 `git push`를 실행하기 전에 최종 커밋 메시지를 보여주고, 사용자가 좌우 방향키로 항목을 이동한 뒤 Enter로 선택할 수 있도록 정확히 다음 네 가지 선택지를 제공합니다:

1. 커밋 후 푸시
2. 커밋만 진행
3. 취소
4. 추가 요구사항 입력

- 기본 선택값은 `1. 커밋 후 푸시`로 설정합니다.
- 사용자가 선택 항목을 변경하지 않고 Enter를 누르면 `커밋 후 푸시`로 처리합니다.
- 사용자가 `1. 커밋 후 푸시`를 선택하면 추가 확인을 요청하지 않고 즉시 `git commit`을 실행합니다.
- `git commit`이 성공한 경우에만 추가 확인을 요청하지 않고 즉시 `git push`를 실행합니다.
- `git commit`이 실패하면 `git push`를 실행하지 않고 실패 내용을 사용자에게 알립니다.
- 사용자가 `2. 커밋만 진행`을 선택하면 추가 확인을 요청하지 않고 `git commit`만 실행합니다.
- 사용자가 `3. 취소`를 선택하면 `git commit`과 `git push`를 모두 실행하지 않습니다.
- 사용자가 `4. 추가 요구사항 입력`을 선택하면 사용자의 추가 요구사항을 반영해 커밋 메시지를 수정하고 다시 확인을 요청합니다.

## 출력 규칙

- 최종 커밋 메시지 자체를 표시할 때는 마크다운, 코드 블록, 따옴표, 설명, 대안, 불필요한 추가 문구를 포함하지 않습니다.
