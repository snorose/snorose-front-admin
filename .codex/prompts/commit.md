현재 저장소의 `.husky/commit-msg`를 읽고 해당 커밋 메시지 규칙을 사용합니다.

## 분석 범위

- 스테이징된 변경사항만 분석합니다.
- 판단 기준은 `git diff --cached --stat`와 `git diff --cached`입니다.
- 스테이징된 변경사항이 없다면 정확히 다음 문구만 출력합니다:

스테이징된 변경사항이 없습니다.

## 커밋 메시지 작성

스테이징된 변경사항이 있다면 `.husky/commit-msg`를 만족하는 커밋 메시지 한 줄을 작성합니다.

- `tag: message` 형식을 사용합니다.
- `.husky/commit-msg`에서 허용하는 태그만 사용합니다.
- 스테이징된 변경사항에 가장 정확히 맞는 태그를 선택합니다.
- 메시지는 간결하되 변경 대상과 핵심 의도가 드러나도록 구체적으로 작성합니다.
- `수정`, `정리`, `개선`, `변경`처럼 넓은 표현만 사용하지 말고, 무엇을 어떻게 바꿨는지 함께 작성합니다.
- 스테이징된 diff에서 핵심 파일명, 기능명, UI 영역, 동작 변화를 우선 반영합니다.
- 작업 내용이 많다면 모든 변경을 나열하지 말고, 사용자에게 가장 중요한 변경이나 대표 목적을 중심으로 한 줄로 요약합니다.
- 메시지가 길어질 때는 장황하게 늘리지 말고, 한 줄 안에서 구체성을 우선합니다.
- 이 저장소의 커밋 기록에 맞춰 메시지 본문은 한국어를 우선 사용합니다.
- 사용자가 `FOCUS`를 제공했다면 추가 맥락으로만 사용합니다: $FOCUS

예시:

- 나쁜 예: `modify: 프롬프트 수정`
- 좋은 예: `chore: 커밋 프롬프트 메시지 작성 기준 구체화`

## 실행 방식 확인

`git commit` 또는 `git push`를 실행하기 전에 최종 커밋 메시지를 먼저 그대로 출력한 뒤, 사용자가 터미널 패널에서 좌우 방향키로 항목을 이동하고 Enter로 선택할 수 있는 TUI를 한 번만 실행합니다.

- Codex 채팅의 일반 텍스트 질문이나 `request_user_input` 선택 UI를 사용하지 않습니다.
- 반드시 `exec_command`의 `tty: true` 세션에서 터미널 TUI를 실행합니다.
- 선택지는 한 줄에 가로로 배치하고, 현재 선택 항목이 시각적으로 구분되도록 표시합니다.
- 사용자가 직접 방향키와 Enter로 선택할 수 있게 터미널 입력을 기다립니다.
- TUI를 실행할 수 없는 환경이라면 `git commit`과 `git push`를 실행하지 말고, 방향키 선택 TUI를 제공할 수 없다고만 알립니다.

정확히 다음 네 가지 선택지를 제공합니다:

1. 커밋 후 푸시
2. 커밋만 진행
3. 취소
4. 추가 요구사항 입력

- 기본 선택값은 `1. 커밋 후 푸시`로 설정합니다.
- 사용자가 `1. 커밋 후 푸시`를 선택하면 이 선택을 `git commit`과 `git push` 모두에 대한 최종 승인으로 간주합니다.
- `1. 커밋 후 푸시`를 선택한 뒤에는 `git commit` 실행 전이나 `git push` 실행 전에 사용자에게 다시 확인하지 않습니다.
- 권한 승인 또는 실행 승인 UI가 필요한 환경이라면 `git commit`과 `git push`를 따로 묻지 말고, 두 작업을 함께 설명해 한 번만 승인 요청합니다.
- 승인되면 즉시 `git commit`을 실행하고, `git commit`이 성공한 경우에만 즉시 `git push`를 실행합니다.
- `git commit`이 실패하면 `git push`를 실행하지 않고 실패 내용을 사용자에게 알립니다.
- 사용자가 `2. 커밋만 진행`을 선택하면 추가 확인을 요청하지 않고 `git commit`만 실행합니다.
- 사용자가 `3. 취소`를 선택하면 `git commit`과 `git push`를 모두 실행하지 않습니다.
- 사용자가 `4. 추가 요구사항 입력`을 선택하면 사용자의 추가 요구사항을 반영해 커밋 메시지를 수정하고 다시 확인을 요청합니다.

## 출력 규칙

- 최종 커밋 메시지 자체를 표시할 때는 마크다운, 코드 블록, 따옴표, 설명, 대안, 불필요한 추가 문구를 포함하지 않습니다.
