---
description: 저장소 PR 템플릿과 라벨 규칙에 맞춰 PR을 작성하고 생성합니다
argument-hint: [BASE="대상 브랜치"] [FOCUS="선택적 맥락"] [DRAFT="true|false"]
---

현재 저장소의 `.github/pull_request_template.md`와 `.github/labeler.yml`을 읽고 PR 작성 규칙으로 사용합니다.

## 사전 확인

- 현재 브랜치를 확인합니다: `git branch --show-current`
- 현재 GitHub 인증 사용자를 확인합니다: `gh api user --jq .login`
- 현재 GitHub 인증 사용자를 확인하지 못하면 PR을 생성하지 말고 인증 상태 확인을 요청합니다.
- 현재 브랜치명에서 이슈 번호 후보를 추출합니다.
  - `#숫자` 패턴을 가장 먼저 사용합니다. 예: `chore/#296-codex` -> `296`
  - `#숫자`가 없다면 브랜치명 안의 첫 번째 숫자 묶음을 보조 후보로 사용합니다. 예: `feat/296-filter` -> `296`
  - 이슈 번호 후보가 여러 개라면 첫 번째 후보만 사용합니다.
- 사용자가 `BASE`를 제공했다면 대상 브랜치로 사용합니다: $BASE
- 사용자가 `BASE`를 제공하지 않았다면 `gh repo view --json defaultBranchRef --jq .defaultBranchRef.name`로 기본 브랜치를 확인해 대상 브랜치로 사용합니다.
- PR 분석 기준은 `git diff <base>...HEAD --stat`, `git diff <base>...HEAD`, `git log <base>..HEAD --oneline`입니다.
- 대상 브랜치와 현재 브랜치 사이에 PR로 만들 변경사항이 없다면 정확히 다음 문구만 출력합니다:

PR로 만들 변경사항이 없습니다.

## PR 제목 작성

`.github/labeler.yml`의 라벨 규칙은 PR 제목을 기준으로 동작합니다. 자동 라벨이 붙도록 PR 제목은 반드시 라벨 규칙에 맞는 `tag: message` 형식으로 작성합니다.

- `.github/labeler.yml`에서 허용하는 제목 prefix만 사용합니다.
- 변경사항에 가장 정확히 맞는 prefix를 선택합니다.
- 제목은 간결하되 변경 대상과 핵심 의도가 드러나도록 구체적으로 작성합니다.
- `수정`, `정리`, `개선`, `변경`처럼 넓은 표현만 사용하지 말고, 무엇을 어떻게 바꿨는지 함께 작성합니다.
- 작업 내용이 많다면 모든 변경을 나열하지 말고, 사용자에게 가장 중요한 변경이나 대표 목적을 중심으로 한 줄로 요약합니다.
- 이 저장소의 커밋 기록에 맞춰 제목 본문은 한국어를 우선 사용합니다.
- 사용자가 `FOCUS`를 제공했다면 추가 맥락으로만 사용합니다: $FOCUS

예시:

- 나쁜 예: `modify: 작업 수정`
- 좋은 예: `feat: 시험후기 신고 상태 필터 추가`

## 이슈 번호 작성

브랜치명에서 추출한 이슈 번호 후보를 PR 본문의 `Close #`에 반영합니다.

- 이슈 번호를 찾았다면 `Close #<issue-number>` 형식으로 작성합니다.
- 예: 현재 브랜치가 `chore/#296-codex`라면 `Close #296`으로 작성합니다.
- 이슈 번호를 찾지 못했다면 `Close #` 형태를 유지합니다.
- 추출한 번호가 실제 이슈 번호인지 애매하면 PR 생성 전에 보여주는 본문 요약에 포함해 사용자가 확인할 수 있게 합니다.

## PR 본문 작성

`.github/pull_request_template.md`의 섹션 구조를 유지해 PR 본문을 작성합니다.

- `Close #`는 브랜치명에서 추출한 이슈 번호를 우선 사용해 채웁니다. 알 수 없다면 `Close #` 형태를 유지합니다.
- `Figma`와 `Notion` 링크는 알 수 있을 때만 채웁니다. 알 수 없다면 빈 링크 형식을 유지합니다.
- `🎯 변경 사항`에는 핵심 변경사항을 구체적인 bullet로 작성합니다.
- 작업 내용이 많다면 모든 변경을 나열하지 말고, 리뷰와 이해에 중요한 변경을 우선 작성합니다.
- `📸 스크린샷`은 UI 변경이 있거나 사용자가 이미 이미지를 제공한 경우에만 채웁니다. 없으면 템플릿의 선택 섹션을 유지합니다.
- `🧐 리뷰어가 어떤 부분에 집중해야 할까요?`에는 리뷰어가 확인해야 할 동작, 경계 조건, 영향 범위를 중심으로 작성합니다.

## Assignee 설정

- assignee는 `gh api user --jq .login`으로 확인한 현재 GitHub 인증 사용자로 설정합니다.
- 특정 사용자명을 하드코딩하지 않습니다.
- 현재 GitHub 인증 사용자를 확인하지 못하면 `gh pr create`를 실행하지 않습니다.

## 자동 라벨 규칙

`.github/labeler.yml` 기준으로 PR 제목 prefix와 자동 라벨은 다음처럼 연결됩니다.

- `feat:` -> `✨ Feature`
- `fix:` -> `🐞 BugFix`
- `docs:` -> `📄 Docs`
- `chore:` -> `🧹 Chore`
- `modify:` -> `🪄 Modify`
- `style:` -> `🎨 UI`
- `merge:` -> `❄️ Release`
- `refactor:` -> `🛠️ Refactor`
- `test:` -> `🧪 Test`

규칙:

- PR 제목 prefix로 자동 라벨이 부여되도록 만들고, `gh pr create`에서 `--label`은 직접 사용하지 않습니다.
- 변경사항이 여러 라벨에 걸쳐 있더라도 가장 핵심적인 변경 목적 하나를 기준으로 prefix를 선택합니다.
- 사용자가 추가 라벨을 명시적으로 요청한 경우에만 별도 라벨 적용 방법을 제안합니다.

## PR 생성 방식 확인

`gh pr create`를 실행하기 전에 PR 제목, 본문 요약, 대상 브랜치, 현재 브랜치, assignee를 보여주고 정확히 다음 세 가지 선택지를 한 번만 제공합니다:

1. PR 생성
2. 취소
3. 추가 요구사항 입력

- assignee는 `gh api user --jq .login`으로 확인한 현재 GitHub 인증 사용자로 설정합니다.
- 기본 선택값은 `1. PR 생성`으로 설정합니다.
- 사용자가 선택 항목을 변경하지 않고 Enter를 누르면 `PR 생성`으로 처리합니다.
- 사용자가 `1. PR 생성`을 선택하면 이 선택을 PR 생성에 대한 최종 승인으로 간주합니다.
- `1. PR 생성`을 선택한 뒤에는 `gh pr create` 실행 전에 사용자에게 다시 확인하지 않습니다.
- 권한 승인 또는 실행 승인 UI가 필요한 환경이라면 PR 제목, 대상 브랜치, 현재 브랜치, assignee를 함께 설명해 한 번만 승인 요청합니다.
- 승인되면 PR 본문을 임시 파일로 작성한 뒤 `gh pr create --title "<title>" --body-file "<body-file>" --base "<base>" --head "<current-branch>" --assignee "<current-github-user>"`를 실행합니다.
- 사용자가 `DRAFT="true"`를 제공했다면 `gh pr create`에 `--draft`를 추가합니다.
- 사용자가 `2. 취소`를 선택하면 `gh pr create`를 실행하지 않습니다.
- 사용자가 `3. 추가 요구사항 입력`을 선택하면 사용자의 추가 요구사항을 반영해 PR 제목과 본문을 수정하고 다시 확인을 요청합니다.

## 출력 규칙

- 최종 PR 제목 자체를 표시할 때는 마크다운, 코드 블록, 따옴표, 설명, 대안, 불필요한 추가 문구를 포함하지 않습니다.
- PR 생성에 성공하면 생성된 PR URL을 사용자에게 알려줍니다.
